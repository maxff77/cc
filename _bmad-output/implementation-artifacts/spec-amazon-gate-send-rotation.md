## Intent

**Problem:** Phase 1 shipped the cookie vault (`gate_cookies` + `cookie_mode` snapshot) but nothing sends or rotates cookies. The checker bot stores the cookie **per Telegram account**, and the deployment shares ONE account across all tenants, so each `.amz <line>` must be immediately preceded by its own `.cookie <value>` with NO other message in between ‚Äî otherwise a cookie leaks across users.

**Approach:** Make the send worker, for a cookie-mode batch, send the atomic pair `.cookie <active_value>` then `.amz <line>` in one worker turn (no `scheduler.pick_next` between them) under the same account-safety guards as a normal send, then HOLD that tenant until the bot's `‚åø Status:` verdict for that line arrives (serialize). The capture consumer classifies the `.amz` reply (Approved/Declined/cookie-dead/format-error) and signals the worker; on a dead verdict the worker rotates to the next active cookie (FIFO) and resends only that line; on exhaustion it pauses the batch (`cookies_exhausted`) so the client adds cookies and resumes from the failed line. `.cookie` is side-band (no `send_log`). This is ONE cohesive engine feature ‚Äî send + rotation + classification are mutually dependent and ship together.

## Boundaries & Constraints

**Always:**
- **Single shared account** ‚Äî respect the constant interval, FloodWait window, and watchdog latch. Never run two `cc-core`. The atomic pair and the per-line serialize protect the cookie-per-account invariant; never interleave another tenant's message (or another line) between `.cookie` and `.amz`.
- **The `.cookie` side-band send runs through the SAME account guards as `.amz`.** Before sending the pair, check `watchdog.is_paused` and `scheduler.flood_remaining()`; if a FloodWait window is open or the watchdog is latched, do NOT send the pair this turn ‚Äî release the line (nothing went out) and re-queue. Wrap the `.cookie` `gateway.send` so `SessionLostError` ‚Üí release line **and** `watchdog.session_lost()` (latch the global pause), and `FloodWaitError` ‚Üí `scheduler.note_flood_wait()` + open the global window + release. Do NOT feed `watchdog.note_sent()` for `.cookie` (it is not a tenant line). A bare unguarded `gateway.send` for `.cookie` is forbidden ‚Äî it would fire inside an open FloodWait window and silently fail to latch the watchdog on session loss.
- **Telethon only in `core/telegram.py`**, `parse_mode=None` load-bearing. The send worker only calls `gateway.send(text)` ‚Äî never imports Telethon, never constructs `.cookie`/`.amz` text anywhere Telethon can see the raw cookie except as the `text` argument. The `.amz` line text already carries `.amz ` from `apply_gate` ‚Äî the worker sends `line.text` verbatim, never re-prepending.
- **Write-ahead + fail-stop for the `.amz` line.** Intent recorded in `send_log` BEFORE the `.amz` send (the existing claim/`record_intent` commit at `send_worker.py:326-328`); `message_id` recorded after, retry-forever (`_record_sent`). The `.amz` `message_id` is the sole attribution key **per attempt**. **`.cookie` is side-band: a bare guarded send with NO `send_log`/`batch_line` row** ‚Äî its confirmation reply attributes to nothing and capture drops it (see content-sniff below).
- **`tenant_id` from the session only**, never body/path. The cookie value is read by `(tenant_id, gate_id)` derived from the claimed line's batch. `gate_id` is a snapshot-only nullable `Batch.gate_id` set at cookie-mode batch creation (no FK ‚Äî history survives gate edits, like the existing `gate_value`/`gate_name` snapshots); never re-resolve gate_id from `gate_value` at send time (a retired+recreated value mis-keys cookies across gate generations).
- **`FOR UPDATE` on contested state** ‚Äî the live batch, the capture session, and the active-cookie selection. `get_active_for_rotation` selects the oldest `status='active'` row by `id ASC` `FOR UPDATE SKIP LOCKED`; `mark_dead` commits the dead status BEFORE the next active row is selected for the resend (or both run in one txn that marks dead THEN selects the next by id ASC excluding the just-marked id) so a just-dead cookie can never be re-picked. Partial unique indexes (`uq_batches_one_live_per_tenant`, ‚â§1 active capture-session) stay intact; `cookies_exhausted`/`verdict_timeout` are `STATE_PAUSED` + a `pause_reason`, never new states.
- **Per-attempt attribution on rotation/timeout-resend.** A re-sent line is a NEW `.amz` message with a NEW `message_id`. Do NOT let the resend overwrite the dead attempt's `send_log` row: BEFORE the rotation resend, persist the dead attempt's terminal `kind='full'` revision (so attribution can resolve its later edits via the prior-responses-row path) AND reset the line's write-ahead intent (clear `message_id` / re-`record_intent`) under the batch `FOR UPDATE` lock so the reused row carries the NEW send's `message_id`. Completa shows the line once via latest-revision-per-`message_id` across attempts; no orphaned `send_log.message_id` may remain unattributable.
- **Every verdict that consumes or rotates a line persists a durable `kind='full'` row** (legacy parity: rejected revisions ARE Completa rows). Approved ‚Üí `status=ok`; Declined ‚Üí `status=rejected`; cookie-dead ‚Üí a full row recording the dead-verdict revision; format-error ‚Üí the line is marked `failed` (a line-state terminal, no rotation). This satisfies `_answered_full_exists` so the 45s reply reconciler stops re-feeding the verdict ‚Äî without it, Declined treadmills forever and a replayed cookie-dead verdict falsely re-rotates a healthy cookie into spurious exhaustion. Only a pure ‚è≥/no-verdict writes nothing.
- **Verdict classification is idempotent and attempt-fenced.** The cookie-mode capture branch is a hard short-circuit that fully OWNS status (it replaces, never augments, the legacy ‚úÖ/‚ùå glyph derivation ‚Äî `Declined`/`Approved` carry no glyph and would otherwise fall through to `previous_status`). Classify ONLY inside the `attributed is not None` branch ‚Äî an unattributed `‚åø Status:` reply is bucketed unmatched exactly like today, never rotated. The capture‚Üíworker signal carries the `.amz` `(chat_id, message_id)` it is answering; the worker accepts a verdict only if it matches the `message_id` it is currently awaiting (stored on the batch alongside `awaiting_verdict_until`) and `awaiting_verdict_until` is still set, verified in-txn under the batch `FOR UPDATE` ‚Äî a verdict for a superseded attempt or an already-cleared await is logged and dropped (closes verdict-edit double-fire, timeout-then-late-verdict, and pause-race stale signals).
- **The side-band cookie confirmation is dropped by a content sniff, NOT by attribution-miss.** A top-of-`process_incoming` literal sniff for the `almacen√≥ tu cookie correctamente` marker returns immediately: no `attribution.resolve`, no 3-attempt unmatched retry, no `_unmatched_total` bump, no `alerts.note_unmatched()`. It MAY feed `watchdog.note_reply()` liveness at enqueue (unchanged), but MUST NOT feed the unmatched/ban-guardrail counter ‚Äî every cookie-mode line yields one confirmation, so the attribution-miss path would steadily inflate the unmatched bucket and trip the guardrail.
- **Capture/legacy parity.** CC dedup is per-session and DB-enforced (`uq_responses_session_cc`, 600-char truncation) ‚Äî never moved into code; the Approved whole-line fallback inherits that truncation.
- **The cookie value is a credential** ‚Äî never logged, echoed, or interpolated into any error/event. Log only `(tenant_id, gate_id, cookie_id, 'MASKED')`. A `.cookie` send failure emits ONLY a masked, value-free log line and routes as release ‚Äî it must NOT emit the per-attempt `error`/`send_error` event (which interpolates `str(e)` and could echo the outgoing text).
- **Migrations before restart** ‚Äî the new columns ship nullable/`server_default` and `alembic upgrade head` runs before the service restart.
- **Rotation:** FIFO by `GateCookie.id ASC` over `status='active'`; on a dead verdict set `status='dead'` (kept for UI, excluded from rotation) and advance. On exhaustion (no active cookie) pause `cookies_exhausted`; resume re-queues and continues from the failed line.

- **`Approved`-reply CC extraction ‚Äî real sample locked.** An Approved reply is one line of the shape: `‚òá CC: 377481016137504|05|2033|3845‚åø Status: Approved ‚úÖ‚åø Response: Tarjeta vinculada correctamente. | Removed: ‚úÖ Removido` ‚Äî the `CC:` card and the `‚åø Status:` verdict sit on the SAME line with NO newline between them. - **Verdict timeout ‚Äî 90s, retry-once, then pause (confirmed).** `awaiting_verdict_until = func.now() + interval '90 seconds'`; on elapse retry the line ONCE with a fresh-picked cookie + a NEW awaited `message_id`; still silent ‚áí pause `verdict_timeout` + owner alert. Hardcoded (pipeline internals are not configurable).

**Never:**
- No in-flight pipelining or rewind of already-sent lines (owner chose serialize-and-wait deliberately). One line's pair ‚Üí wait ‚Üí next line.
- No second Telegram account; no Telethon outside `core/telegram.py`.
- No edits to legacy `app.py` / `core.py` / `static/`.
- Do NOT weaken rate-limiting, the FloodWait window, or the watchdog. Do NOT break the Phase-1 vault (`/api/cookies`, masking, dedup) or the `special_mode` capture path (which stays byte-for-byte unchanged).
- No `send_log`/`batch_line` row for `.cookie`. Never persist the `[„Éæ] ‚Ä¶ almacen√≥ tu cookie correctamente. ‚úÖ` confirmation as a response/CC row.
- Do NOT retry the bare `.amz` alone after a mid-pair FloodWait ‚Äî once the `.cookie` is out and the global window opens, the cookie context on the account is burned; the whole pair must re-send fresh.

## Spec Change Log

### 2026-06-19 ‚Äî review loopback (3 bad_spec re-implemented, 3 patches applied, 1 deferred)

The first impl identified "the line awaiting a verdict" by **line state** (`LINE_SENT`) instead of the **attempt-fence** (`Batch.awaiting_message_id`). The fence is the single source of "which line is in-flight" ‚Äî re-implemented around it.

- **Finding (blind+edge+acceptance, HIGH):** multi-line cookie batches re-sent already-consumed lines. A consumed (approved/declined) line stays `LINE_SENT`, so with a second in-flight line there are TWO `LINE_SENT` rows; `_sweep_verdict_timeouts` and `requeue_failed_cookie_line` filtered `WHERE state==LINE_SENT` (`.first()`/`.all()`) and re-sent finished lines (duplicate `.amz`/Completa/CC/charge). **Amended:** both resolve the awaited line via the fence ‚Äî `send_log.(chat_id, message_id) == batch.(awaiting_chat_id, awaiting_message_id)` under the held `FOR UPDATE` ‚Äî and operate on that single `line_id` only (no-op when the await is cleared, e.g. `cookies_exhausted` where the line is already `queued`). **Known-bad avoided:** duplicate sends/charges on every multi-line Amazon batch. **KEEP:** serialize, the DB-`now()` gate, per-attempt `send_log` reuse.
- **Finding (blind+acceptance+edge, MED):** `cookie_dead` rotation re-derived the dead cookie as "oldest active" across two UNLOCKED sessions (not the cookie actually sent), and `BatchLine.failed_cookie_id` was declared but never written/read (dead schema); a concurrent cookie DELETE during the 90s await could burn a healthy cookie. **Amended:** the worker writes `BatchLine.failed_cookie_id = <sent cookie id>` at send time; on a `cookie_dead` verdict, in ONE txn under the batch `FOR UPDATE`: `mark_dead(that id)` ‚Üí flush ‚Üí `count_active_for` / next pick (the dead row now excluded by the `status` filter) ‚Üí re-queue or pause `cookies_exhausted`. **Known-bad avoided:** marking a healthy cookie dead / false exhaustion. **KEEP:** FIFO `id ASC`, `mark_dead` idempotent.
- **Finding (edge, MED):** a manual client pause DURING a verdict-await left the gate armed (silent stall until the 90s timeout) and `_apply_verdict`/`_drain_verdicts` had no `state==SENDING` guard, so a verdict mutated a paused batch. **Amended:** pausing a cookie-mode batch with a live await clears the await + re-queues the awaited line (so resume re-sends fresh); `_apply_verdict` and the drain are gated on `batch.state == STATE_SENDING` (a verdict for a non-sending batch is dropped; the timeout/resume path recovers the line). **Known-bad avoided:** stalled-after-resume + behind-the-back mutation of a paused batch.
- **Patches applied:** (#5) capture's verdict-signal guard now also requires `reply.reply_to_msg_id is not None` (never signal an un-fenceable `message_id=None`); (#6) the cockpit renders a `verdict_timeout` pause notice (mirrors the exhausted notice, no CookieManager); (#7) `awaiting_reply_count` counts DISTINCT `line_id` among full rows so a multi-attempt (rotated) line counts once.
- **Deferred (‚Üí deferred-work.md):** (#4) the `_timeout_retried` set is process-memory, so a crash-loop around the 90s timeout re-grants the retry-once budget per restart (unbounded resends). Low ‚Äî needs a crash loop; a durable fix needs a new column. Documented for a follow-up.

## Design Notes

**Serialize gate lives in SQL, not the scheduler's clock.** `awaiting_verdict_until` is a Postgres `datetime`; the scheduler runs on `time.monotonic()` (`scheduler.py:46`). Comparing them is meaningless. Resolve the skip in `active_senders` (`AND (awaiting_verdict_until IS NULL OR awaiting_verdict_until <= now())`) so DB `now()` is the single time source ‚Äî survives restart, no clock-mixing, and an awaiting batch (still `state='sending'` with a queued line) is simply not returned.

**Capture‚Üíworker signal (not the client WS).** The verdict hand-off is engine-internal: capture tells the single send-worker task to rotate/clear via an in-process async signal (same pattern as the capture queue/scheduler cursor ‚Äî process memory, reset on restart), keyed by the `.amz` `(chat_id, message_id)` plus the resolved `line_id`. `awaiting_verdict_until` + the awaited `message_id` (stored on the batch) are the DURABLE, authoritative gate; the signal is the fast path. The worker drops any signal not matching the currently-awaited `message_id` or whose await is already cleared (verified in-txn under the batch `FOR UPDATE`). Do NOT route through `broadcaster` ‚Äî that is reserved for tenant-scoped client fan-out (the `cookies_exhausted` prompt), not capture‚Üíworker control flow.

**Per-attempt attribution.** A dead cookie's `.amz` already went out with a committed `message_id`. The resend is a DIFFERENT `message_id` for the SAME `line_id`, but `set_message_id` is a bare UPDATE keyed on `line_id` ‚Äî so before resending, the worker (1) persists the dead attempt's terminal `kind='full'` row (its later edits then resolve via attribution's prior-responses-row path keyed on `(chat_id, message_id)`, surviving the `send_log` overwrite) and (2) resets the line's intent so the reused row carries the NEW `message_id`. Latest-revision-per-`message_id` keeps Completa at one visible row.

**gate_id snapshot.** `Batch` deliberately has no FK to `gates` (history survives gate edits ‚Äî `attribution.py` backfill keys on the value snapshots). The active cookie is keyed by `(tenant_id, gate_id)`, so snapshot a nullable `Batch.gate_id` at cookie-mode batch creation (gate_id is in hand there as `body.gate_id`/`gate.id`). Do NOT re-resolve from `gate_value` at send time ‚Äî a retired+recreated value mis-keys cookies across gate generations.

**Classification fork mirrors special_mode but OWNS status.** At `capture.py:318` the `special` branch shows the idiom: read the snapshot, parse a marker, derive status. The cookie-mode branch is parallel but a HARD short-circuit ‚Äî `Approved`/`Declined` carry no ‚úÖ/‚ùå glyph, so it must replace (not fall through to) the glyph elif-chain or status mis-derives to `previous_status`. Parse runs on the REDACTED text (`_tidy_separators` already rewrote separators in `redact_reply_text` before any fork), anchored to `‚åø Status:` with a separator/whitespace-tolerant regex; check `Status` before `Format`.

**Atomic pair = one worker turn, guard-gated up front.** The pair is sent inside `step()` after the claim commit, before yielding, with watchdog/FloodWait checked BEFORE the `.cookie`. A pause/stop landing BETWEEN `.cookie` and `.amz` releases the line (only `.cookie` went out ‚Äî its confirmation is sniffed away) and re-queues; resume picks a fresh cookie. A FloodWait on the `.amz` is a PAIR-ABORT, not a bare-`.amz` retry: once the global window opens another tenant's `.cookie` could clobber the account, so the whole pair re-sends fresh on resume. This ‚Äî plus gating `.cookie` on the global window ‚Äî is what makes "no other message between this tenant's `.cookie` and its `.amz`" actually hold.
