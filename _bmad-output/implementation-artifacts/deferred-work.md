## Story 2-3 review (deferred)

- MEDIUM — `backend/app/core/send_worker.py:50`: `wake()` is one global event consumed by whichever sleep the lone worker is in; 2.3 adds the first callers (pause/resume/stop in `api/batches.py`). Cross-tenant collateral: (1) tenant A's control cuts tenant B's in-flight FloodWait sleep — the re-check sees B still 'sending' and retries `gateway.send` on the SHARED account before Telegram's window elapsed, typically earning another (escalating) FloodWait; (2) the same wake cuts the inter-send interval sleep in `run_worker`, so alternating pause/resume drives the global send rate past the system interval (FR12 pacing bypass) for everyone. Fix: deadline loop — record `deadline = now + seconds` before `sleep_cancelable`; after a wake-interrupted sleep, if the re-read state is still 'sending' (wake was for another batch) re-sleep the remainder instead of retrying; same remainder re-sleep for the post-send interval (pause still lands instantly since a paused batch isn't served). Legacy pause→resume-retries-early semantics survive via release/re-claim.
- MEDIUM — `frontend/components/client-nav.tsx:34`: the 'En pausa' pill is unreadable — `PILL_CLASS.paused` uses `bg-warning/18 text-warning-foreground`, but `--warning-foreground` is near-black (contrast color for SOLID warning fills) and the app hardcodes dark mode, so it renders near-black 10px text on a near-black amber tint. Siblings do it right (`text-accent`, `text-danger`). Fix: `bg-warning/18 text-warning`.
- LOW — `frontend/components/batch/flood-notice.tsx:20`: the 1s `setInterval` never stops after the countdown expires while `floodUntil` stays non-null — `flood.wait` is GLOBAL but the clearing signals are tenant-scoped to the sender, so an idle tenant's store keeps a stale `floodUntil` and the interval re-renders every FloodNotice consumer once per second until a reconnect snapshot resets it. Fix: clear the interval at expiry (e.g. `if (Date.now() >= until) window.clearInterval(id)`) or a store action that nulls `floodUntil` on expiry so the existing `=== null` early-returns kick in.
